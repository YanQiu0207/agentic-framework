# 文章总结：Harness Engineering 如何工程化落地

> 来源：《万字干货！Harness Engineering 如何工程化落地？》，白家杰（腾讯云开发者）。
> 样本工程：JK Launcher——一个 WPF / .NET Framework 桌面启动器（面向 Unity 研发流程），全程由 AI 写代码、人只负责搭 Harness。

## 1. 核心论点

Harness Engineering 不是某个工具，也不是某条提示词技巧，而是**一整套让 AI 在工程里稳定产出正确结果的工程系统**。它要解决的不是「让 AI 更聪明」，而是把 AI 从一个会临场发挥的模型，变成「可约束、可协作、可校验、可持续维护的执行系统」。（原文第 1 章、第 12 章）

三个关键词：

- **稳定**：不是这次运气好，而是换需求、换维护人后仍能工作。
- **产出**：不止代码，还包括需求、方案、验证、交付等全过程产物。
- **正确**：不是「做完就算」，而是有办法判断到底做得对不对。

## 2. 六个构件

| 构件 | 回答什么问题 | 在工程里的角色 |
| --- | --- | --- |
| Rule | 什么事绝对不能乱来 | 红线、底线（软约束） |
| Skill | 这件事具体怎么做 | 固定动作的标准操作手册 |
| Sub Agent | 复杂任务谁来分工 | 不同阶段的专业角色 |
| Workflow | 角色按什么顺序接力 | 前进 / 暂停 / 打回 / 重跑规则 |
| Scripts | 谁判断到底做没做好 | 统一门禁与事后验证（硬约束） |
| MCP | AI 怎么安全接外部系统 | 外接能力接口（CI / 签名 / 发布等） |

文章反复强调一个递进关系：**Rule 是软约束（会被忽略、绕过、解释性执行），能判定的约束最终都应下沉为 Scripts 这种硬门禁。**（第 3 章、第 7.4 节、第 8 章）

## 3. 四块拼图（全局视角）

| 拼图 | 管什么 | 对应章节 |
| --- | --- | --- |
| 约束与流程 | AI 按什么顺序、什么边界做事；协作时谁接哪一棒 | 第 3～7 章 |
| 反馈 | 做完后系统能否对结果有说法（而非「模型觉得做完了」） | 第 8 章 |
| 知识库 | 项目级、写进仓库的「从哪进门」索引（非整仓百科） | 第 9 章 |
| 进化 | 规范往哪沉淀、人退到哪一层、Harness 如何持续变厚 | 第 10～11 章 |

四块缺一不可，典型症状：只有流程没反馈 → 完成幻觉；只有反馈没流程 → 上游一团即兴；没知识库 → 重复造轮子；没进化 → 冻结在某一版。（第 12.6 节）

## 4. 七个 Agent（结构化调度）

文章在三条多 Agent 路线里做了选型（第 5 章）：

| 路线 | 代表 | 结论 | 理由 |
| --- | --- | --- | --- |
| 去中心化协商 | AutoGen GroupChat 式 | 放弃 | 路径不稳定、责任不清、难维护 |
| 动态生成 Agent | 只固定 PM，临时招聘 | 放弃 | 角色漂移、维护成本高 |
| 固定角色 + 固定流程 | 中央 PM 路由 + 7 个定死角色 | **选用** | 研发大阶段本就固定，稳定 / 可审计优先 |

最终收敛出的 7 个角色（第 6 章）：

| # | 角色 | 解决什么 |
| --- | --- | --- |
| 1 | 项目经理（PM） | 整条链怎么有序串起来（只路由，不做专业判断） |
| 2 | 需求分析 | 想做什么（模糊诉求 → 清晰需求） |
| 3 | 方案设计 | 打算怎么做（需求 → 技术方案） |
| 4 | 闸门总控（可行性） | 现在能不能做（开发前最后把关） |
| 5 | 开发实现 | 真正把它做出来 |
| 6 | 代码审查 | 是不是按要求做出来（实现质量、需求 / 方案一致性） |
| 7 | 测试验证 | 做出来的东西到底能不能用（功能 / 边界 / 回归 / 稳定性） |

落地形态：中央 PM 作为 Orchestrator，按 Workflow 依次调用子 Agent；Agent 间靠**文档链**传递状态（每棒把产出写成文档，下一棒读上一棒的文档），而非框架消息总线。（第 7.8 节配图说明）

## 5. 关键落地机制

### 5.1 总验证脚本（Scripts，文章眼中最关键的基础设施，第 8 章）

一套「研发后验证总入口」，把散落在 Rule 里的约束落成可执行检查，客观判定「这次开发算不算完成」。三类检查：

| 类别 | 内容（以 WPF / .NET 项目为例） |
| --- | --- |
| A 类·静态规范 | XAML 中文 / Emoji、C# 8.0+ 语法、`MessageBox.Show`、硬编码 UI 文案、日志前缀、`last_state.json` 直接访问、SVN 认证参数、文件超长、本地化合规 |
| B 类·交付门槛 | 编译必过、测试全过、**测试数量不能异常减少** |
| C 类·工程一致性 | 规则文件多格式同步、`.cs` 是否都进了 `.csproj` |

价值：AI 的「完成」定义从「我觉得做完了」变成「脚本判定通过才算做完」——这是质变。

### 5.2 基线对比（第 7.5 节）

改代码**前**跑一次验证存为基线，改**后**再跑一次，只追究**新增**的失败 / 告警 / 违规。专治 AI 用「这是历史遗留、不是我引入的」甩锅——是不是你引入的，靠两份报告的 diff 说话，不靠嘴。「测试数量异常减少」也是它的变体（抓偷删测试）。

### 5.3 dev-map（开发导航地图，第 9.2 节）

由**开发 Agent** 维护，规则是「改代码前先查 dev-map」。记的是代码级落点：某功能落在哪些文件、某类配置在哪定义、改一个模块牵动哪些链路、既有的标准写法长什么样。作用是让 AI 别在已经长好的城市里重复铺路。仓库大时拆成多本薄册。

### 5.4 任务看板（第 9.3 节）

由 **PM** 维护，是项目级任务总览：在做哪些任务、各到哪阶段、文档目录在哪、已完成任务的交付结论。新需求进来时先看一眼，避免「新需求把旧设计冲掉」。

### 5.5 Workflow 三层（第 7.6 节）

把散在 PM 长文里的流程抽成可维护资产：① 流程定义文件（阶段 + 迁移边 + 回退边）；② 角色契约（每个角色读什么 / 写什么 / 何时阻塞）；③ 流程校验脚本（查契约是否齐、角色是否对得上）。

### 5.6 模型分档（第 6.6 节）

PM 这类只做路由的角色用便宜模型；需求 / 方案 / 审查 / 测试这类专业判断重的角色用强模型。把算力花在刀刃上。

## 6. 四条工程纪律

1. **下游不改上游**：下游觉得上游产出不合格，只能提阻塞项，由 PM 正式打回，不能自己顺手改。（第 7.1 节）
2. **PM 只路由不越界**：不写需求、不定方案、不改代码、不替专业角色下结论。（第 6.4、7.2 节）
3. **上下文纪律**：每一棒只给当前该看的材料，不一次性把规则 / 地图 / 历史全堆给 AI。（第 1.5 节、第 9 章）
4. **Memory 靠边站**：团队要对齐的真相进仓库（能判定就进 Scripts，其次 Rule），不靠聊天记忆；「错题集」要晋级为 Rule / Scripts。（第 10 章）

## 7. 最小落地顺序（第 7.8 节）

1. 先磨一份能指导开发的 SPEC（无模糊词）
2. 补最关键的 Rule（别贪多）
3. 高频固定动作下沉为 Skill（编译 / 测试 / 验证）
4. 单 Agent 失稳时再拆多 Agent
5. 多 Agent 变复杂时再补流程定义文件 + 角色契约
6. 进入持续迭代时再补 dev-map + 任务看板
7. 要把闭环往外推时再接 MCP

口诀：先让 AI 知道做什么 → 再知道必须怎么做 → 再学会分工 → 最后让流程本身变成可维护资产。

## 8. 平台与文章局限

- **平台**：主要在 Cursor 落地（含按 Agent 配不同模型）；提到 CodeBuddy；LangGraph、MCP 是未来方向。
- **局限**：文章是「设计 + 演化复盘」，几乎没给实现细节——目录结构、流程定义文件格式、总验证脚本代码、Agent 间产物传递机制都未展开。**它给的是 why，不是 how 的代码级细节。**
